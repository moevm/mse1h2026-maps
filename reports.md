## О проекте
### Постановка задачи
#### Цель проекта
Разработать приложение для построения интерактивных карт знаний по заданной теме, ориентированное на образовательный контекст.

#### Проблема
В процессе обучения часто требуется наглядно изобразить структуру изучаемой области, ключевые понятия и их взаимосвязи. Требуется удобный инструмент, который бы собирал информацию из открытых баз знаний, научных публикаций и веб‑ресурсов и визуализировал её с возможностью редактирования.

### Требования
#### Функциональные требования

1. Управление темой карты
   - пользователь должен иметь возможность ввести тему (ключевое слово или фразу) для генерации карты;
   - система должна запустить процесс сбора данных по указанной теме.
2. Сбор данных
   - модуль сбора данных должен обращаться к следующим категориям источников:
     - открытые базы знаний: Wikidata, DBpedia, Freebase, YAGO, Linked Open Data Cloud;
     - научные базы: Semantic Scholar, Microsoft Academic Graph, Google Scholar, Crossref;
     - веб-ресурсы: Stack Overflow, GitHub;
     - генеративные модели: возможность отправлять запросы к LLM для получения дополнительных связей или объяснений.
3. Структурирование данных
   - приложение должно анализировать собранные данные и выделять ключевые узлы (понятия, сущности) и отношения между ними.
4. Интерактивность карты
   - должна быть функция видимости по источникам информации: пользователь может выбрать, данные из каких источников скрывать или показывать;
   - должна быть функция видимости по типу связи: пользователь может выбрать, какие связи скрывать или показывать.
5. Визуализация
   - карта должна отображаться на веб-странице;
   - должны быть реализованы функции взаимодействия: масштабирование, перемещение, выделение узлов, просмотр информации о связи.
6. Хранение данных
   - промежуточные результаты сбора хранятся в PostgreSQL;
   - итоговая структура графа (узлы, связи, атрибуты) хранится в Neo4j для эффективной обработки и запросов.

#### Нефункциональные требования

Должны выполняться требования ко всем слоям прозрачности.

#### Технологии

1. Python+Django для серверной части.
2. JS для интерактивной визуализации.
3. PostgreSQL для временного хранения собранных данных.
4. Neo4j для хранения графа.


### Сценарии использования
#### 1. Создание новой графовой карты по теме
| | |
|---|---|
| Действующие лица | Пользователь |
| Цель | Запустить процесс генерации карты, указав начальную тему. |
| Предусловие | Пользователь открыл веб-приложение в браузере и открыл меню (боковую панель). |
| Успешный сценарий | 1. Пользователь вводит название темы (слово или словочетание) в поле «Введите ключевые слова».<br> 2. Пользователь нажимает кнопку «Поиск» (значок лупы).<br> 3. Система отправляет запрос на сервер часть.<br> 4. Система получает данные из внешних источников.<br> 5. Система сохраняет узлы и связи в Neo4j и PostgreSQL.<br> 6. По мере работы системы (получения информации от источников) пользователь видит постепенно отрисовывающийся граф.<br> 7. Пользователь видит полностью готовый граф.|
| Альтернативный сценарий | **4а.** Один из внешних источников не отвечает.<br> **4б.** Система продолжает сбор данных из оставшихся источников.<br>|
| Результат | Система по заданной теме нашла информацию, построила граф и вывела его пользователю. |

#### 2. Редактирование карты 
| | |
|---|---|
| Действующие лица | Пользователь |
| Цель | Убрать лишние или ошибочные узлы|
| Предусловие | Карта по теме уже сгенерирована и отображается на экране. |
| Успешный сценарий | 1. Пользователь просматривает интерактивную карту.<br> 2. Пользователь замечает узел, который не относится к теме.<br> 3. Пользователь нажимает на узел правой кнопкой мыши .<br> 4. В контекстном меню пользователь выбирает пункт «Удалить узел».<br> 5. Система запрашивает подтверждение.<br> 6. Пользователь подтверждает удаление.<br> 7. Система удаляет узел и его связи из текущего отображения и из базы данных графа.<br> 8. Система перерисовывает карту. |
| Альтернативный сценарий | **4а.** Вместо удаления пользователь выбирает пункт «Скрыть».<br> **7а.** Система скрывает узел визуально, но сохраняет его в базе на случай, если пользователь захочет восстановить. |
| Результат | Из графа удалены выбранные элементы. |

#### 3. Просмотр деталей узла
| | |
|---|---|
| Действующие лица | Пользователь |
| Цель | Получить подробную информацию об элементе карты и перейти к первоисточнику. |
| Предусловие | Карта по теме уже сгенерирована и отображается на экране. |
| Успешный сценарий | 1. Пользователь кликает по наводит курсор на интересующий узел.<br> 2. Система подсвечивает узел и показывает всплывающую подсказку с краткой информацией (название, тип узла).<br> 3. Пользователь кликает по узлу для просмотра полной информации.<br> 4. В правой боковой панели система отображает:<br>      - общие данные: название, краткое описание;<br>   - информация о сущности: расшифровка, определения;<br>   - ссылки на ресурсы: кликабельные URL внешние источники. |
| Альтернативный сценарий | — |
| Результат | Пользователь видит краткую и полную информацию об узле, а также ссылки на ресурсы. |
